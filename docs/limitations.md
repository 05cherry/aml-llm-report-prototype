# 한계 및 엣지 케이스 (Limitations)

본 프로토타입은 **데모**이며 의도적으로 핵심 기능만 구현했다. 운영 적용
전에 인지해야 할 한계와, 더미 데이터에서 관찰된 엣지 케이스를 정리한다.

## 1. 도메인 기준의 한계

- `docs/aml-signals.md`의 임계값(1,000만원, 6시간, 5,000만원, 3배 등)은 모두
  **시연용 예시값**이며 실제 규제 기준이 아니다. 실제 트래블룰·CTR 기준값으로
  튜닝하는 것은 프로토타입 범위 밖이다.
- 외부 위험 주소 DB(체이널리시스 등) 연계가 없다. `counterparty_risk`는 입력
  데이터에 이미 라벨링되어 있다고 전제한다. 실제로는 이 라벨링 자체가 핵심 난제다.

## 2. LLM 기반 판정의 한계

- **비결정성**: 동일 입력이라도 모델 출력 문장은 변동할 수 있다. 이를 줄이기 위해
  `temperature=0`과 tool 스키마 강제를 쓰지만, 자연어 근거 표현까지 완전히 고정되진
  않는다. 형식(4개 섹션)과 위험도는 코드에서 고정/파생하므로 **구조적 일관성**은 보장된다.
- **위험도는 코드에서 파생**한다. 모델이 제출한 `risk_level`은 렌더링에서 무시하고,
  `derive_risk(signals)`가 `docs/aml-signals.md` 위험도 표를 그대로 적용한다. 모델이
  신호는 맞게 잡고 위험도만 틀리는 경우를 구조적으로 차단한다.
- **오탐 전제**: 신호 탐지 = 의심거래 확정이 아니다. 최종 STR 판단은 준법감시
  담당자의 몫이다. 리포트의 "추가 확인 항목"은 그 1차 보조 자료다.

## 3. 프롬프트 가드와 그 한계

LLM의 과탐지를 줄이기 위해 시스템 프롬프트에 두 가지 일반 규칙을 둔다.

- **(A1)** 소득 구간에 상한이 없으면(`100m_plus`) S5의 '상한 3배' 검사는 평가 불가 →
  그 근거만으로 S5를 판정하지 않는다. (ACC-B가 S5로 과탐지되지 않는 이유)
- **(A2)** S2(통과 계좌)는 입금→출금 구간에 trade가 0건일 때만 성립. (ACC-D가 S2로
  과탐지되지 않는 이유)

한계: 이 가드는 특정 계정/거래 ID를 넣지 않은 **일반 규칙**으로만 표현했다(정답
맞추기식 튜닝 금지). 따라서 문서에 명문화되지 않은 새로운 엣지 케이스는 가드가
포착하지 못할 수 있다. 가드가 필요해지면 먼저 `docs/aml-signals.md`를 고친다.

## 4. 더미 데이터에서 관찰된 엣지 케이스

- **ACC-C의 다중 신호 동시 충족**: 시나리오 의도는 "신규 계정 + 고위험 주소
  (S3+S4)"였으나, 실제 데이터는 입금 직후 97% 출금(S2)과 소득 대비 과도 거래(S5)
  조건도 함께 충족한다. 이는 오탐이 아니라 근거가 모두 타당한 케이스로, 위험도는
  어차피 `높음`(S4 포함 또는 신호 2개 이상)으로 동일하다. 하나의 행위가 여러 신호에
  걸릴 수 있음을 보여준다.
- **정상 계정(ACC-E) 미탐지**: 모든 신호 기준 미충족 → "특이사항 없음"으로 명시
  출력된다. 모든 계정을 의심으로 분류하지 않는 것이 오탐 관리의 핵심이다.

## 5. 범위 밖 (다음 단계)

- 계정 **간** 패턴 미탐지: 분산 이체, 순환 거래, 연계 계정 군집 등 단일 계정 관점으로는
  보이지 않는 패턴은 현재 구조(계정 1개 = 호출 1회)로 탐지하지 않는다.
- 실데이터 적용 시 개인정보 국외이전·망분리, API 비용·레이트리밋, 재시도/백오프,
  대량 처리(배치)는 모두 범위 밖이다.
- 시계열·네트워크 기반 탐지, 외부 위험 주소 DB 연계는 `docs/aml-signals.md`의
  "다음 단계"와 동일하게 향후 과제로 둔다.
